# 요구사항 정의서

**프로젝트명**: roll-my-intro-site
**작성일**: 2026-02-21
**버전**: v1.0.0

---

## 1. 프로젝트 개요

개발자의 포트폴리오 및 자기소개를 담은 싱글 페이지 웹 애플리케이션입니다.
방문자에게 개발자의 역량, 경험, 프로젝트를 효과적으로 전달하는 것을 목표로 합니다.

---

## 2. 기술 스택

| 구분 | 기술 |
|------|------|
| 프레임워크 | Vue 3 (Composition API) |
| UI 프레임워크 | Quasar Framework |

| 패키지 매니저 | npm |
| 배포 | GitHub Pages (gh-pages 브랜치) |

---

## 3. 페이지 구성

싱글 페이지(SPA) 구조로, 스크롤 방식의 섹션 탐색을 제공합니다.

### 3.1 섹션 목록

| 섹션 ID | 섹션명 | 설명 |
|---------|--------|------|
| `hero` | 히어로 | 이름, 직함, 한 줄 소개, CTA 버튼 |
| `about` | 소개 | 프로필 사진, 자기소개 텍스트, 기본 정보 |
| `skills` | 기술 스택 | 보유 기술을 카테고리별로 표시 |
| `experience` | 경력 | 회사명, 재직 기간, 담당 업무 타임라인 |
| `projects` | 프로젝트 | 대표 프로젝트 카드 (썸네일, 설명, 링크) |
| `education` | 학력 | 학교명, 전공, 졸업 연도 |
| `contact` | 연락처 | 이메일, SNS 링크, 연락 폼 |

### 3.2 네비게이션

- 상단 고정 헤더(네비게이션 바)
- 각 섹션으로 앵커 스크롤 이동
- 현재 섹션 활성화 표시
- 모바일: 햄버거 메뉴 → 드로어(Drawer) 방식

## 4. 기능 요구사항

### 4.1 필수 기능 (Must Have)

- [ ] **히어로 섹션**: 이름, 직함, 한 줄 소개, GitHub/이메일 바로가기
- [ ] **소개 섹션**: 개발자 자기소개, 이름/나이/위치/이메일 등 기본 정보
- [ ] **기술 스택 섹션**: 기술을 카테고리(Frontend / Backend / DevOps / Tools)로 구분, 숙련도 표시
- [ ] **경력 섹션**: 타임라인 형식의 경력 사항 표시
- [ ] **프로젝트 섹션**: 카드 형식의 대표 프로젝트 목록 (GitHub 링크, 라이브 링크 포함)
- [ ] **연락처 섹션**: 이메일, GitHub, LinkedIn 등 소셜 링크
- [ ] **반응형 레이아웃**: 모바일(~767px) / 태블릿(768px~1023px) / 데스크톱(1024px~)
- [ ] **부드러운 스크롤 애니메이션**: 섹션 간 스크롤 시 애니메이션
- [ ] **다크/라이트 모드 토글**

### 4.2 선택 기능 (Nice to Have)

- [ ] **학력 섹션**: 학력 사항 표시
- [ ] **스크롤 진행률 표시**: 페이지 상단 프로그레스 바
- [ ] **Back to Top 버튼**: 하단에 상단 이동 버튼
- [ ] **방문자 애니메이션**: IntersectionObserver 기반 스크롤 진입 애니메이션
- [ ] **복사 기능**: 이메일 주소 클립보드 복사

## 5. 비기능 요구사항

### 5.1 성능

- Lighthouse 성능 점수 **85점 이상** 목표
- 초기 로드 시간 **3초 이내** (3G 기준)
- 이미지 최적화 (WebP 포맷, Lazy Loading)

### 5.2 접근성

- WCAG 2.1 AA 수준 준수
- 키보드 네비게이션 지원
- 스크린 리더 호환 (적절한 aria 속성)
- 색상 대비 4.5:1 이상

### 5.3 호환성

- 최신 Chrome, Firefox, Safari, Edge 지원
- 모바일 브라우저(iOS Safari, Android Chrome) 지원
- IE 지원 불필요

### 5.4 유지보수성

- 데이터와 UI 분리: `src/data/profile.ts`에서 콘텐츠만 수정하면 전체 페이지 업데이트
- TypeScript 타입 정의로 데이터 일관성 보장
- 컴포넌트 단위 모듈화

## 6. 브랜치 전략

```
master          ─── 배포용 (GitHub Pages)
develop         ─── 통합 브랜치
  ├── feature/plan    기획 (요구사항, 데이터 구조)
  ├── feature/design  디자인 (레이아웃, 컴포넌트 구조, 스타일 가이드)
  ├── feature/dev     개발 (실제 기능 구현)
  └── feature/test    테스트 (단위/E2E 테스트)
```

---

## 7. 커밋 컨벤션

```
feat: 새로운 기능 추가
fix: 버그 수정
docs: 문서 수정
style: 코드 포맷, 스타일 변경
refactor: 코드 리팩토링
test: 테스트 코드
chore: 빌드, 설정 변경
```

- 커밋 메시지는 **한글**로 작성
- 작업 단계별로 커밋을 나눠서 기록
